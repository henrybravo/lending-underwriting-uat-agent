# Daedalion — Findings from SDK Agent Project

> Written after applying daedalion v0.3.3 to a Copilot SDK agent project
> (`lending-underwriting-uat-agent`). Intended as feedback for the daedalion
> maintainer and as a reference for future projects deciding how to adopt it.

---

## Context

This project runs a mortgage-underwriting UAT agent via the **GitHub Copilot SDK
for Python** (`agent.py`). The agent is driven by skill files loaded at runtime
using the SDK's `skill_directories` parameter — not by the IDE. Daedalion was
already scaffolded in the repo, and we ran `daedalion build` to assess whether
it was still in sync with the code after direct code changes.

---

## Finding 1 — Hub/Spoke Output Breaks in SDK Mode

### What daedalion generates

`daedalion build` produces a **hub/spoke** layout for skills:

```
.github/skills/lending-underwriting/
├── SKILL.md                              # hub: compact requirements + links
├── income-verification.md                # spoke: WHEN/THEN scenarios
├── debt-to-income-calculation.md
├── credit-assessment.md
├── decision-determination.md
├── audit-logging-and-security.md
├── performance-targets.md
└── bonus-commission-income-verification.md
```

The hub (`SKILL.md`) links to spokes via markdown:
```markdown
## Acceptance Criteria
- [Income Verification](./income-verification.md)
- [Debt-to-Income Calculation](./debt-to-income-calculation.md)
...
```

### How this works in the IDE

In VS Code, the agent loads `#lending-underwriting` by following the markdown
links. This is the intended design: SKILL.md is compact and the IDE lazily
follows links when the model needs scenario detail.

### How this breaks in the SDK

The Copilot SDK's `skill_directories` parameter loads **every `.md` file** in
the directory eagerly into the system prompt — it does not follow links on
demand. The result:

| | Chars | ~Tokens/call |
|---|---|---|
| SKILL.md alone (old hand-tuned) | 4,876 | ~1,220 |
| New SKILL.md (hub) | 12,330 | ~3,082 |
| 7 spoke files (all loaded) | 4,001 | ~1,000 |
| **Total (new build)** | **16,331** | **~4,082** |
| Delta vs old | +11,455 | **+~2,860/call, +~8,600/run** |

The spoke files contain the same WHEN/THEN scenarios that SKILL.md already
summarises in its Requirements section — content appears **twice** in the system
prompt. We reverted to the hand-tuned SKILL.md to avoid this overhead.

### Root cause in the code

`src/generators/skill.ts` does not check `agents.target`. Both IDE and SDK modes
produce identical output. The hub/spoke pattern is correct for IDE; for SDK a
flat single-file output (SKILL.md only, scenarios inline) would be correct.

### Recommendation

Add SDK-aware skill generation: when `agents.target: sdk`, either:

- **Option A (preferred)**: emit only `SKILL.md` with scenarios inlined (no
  spoke files). Callers load the directory once; no duplication.
- **Option B**: emit a separate `SKILL.sdk.md` flat file and let `daedalion
  build` write both; the project's `skill_directories` points to the flat one.

## Finding 2 — `copilot-instructions.md` Lost Project Conventions

### What changed

The previous (hand-edited) `copilot-instructions.md` contained the full
`openspec/project.md` content (Tech Stack, Code Style, Architecture Patterns,
Testing Strategy, Constraints). After `daedalion build` it became a 12-line
stub:

```markdown
# Copilot Instructions
> Auto-generated by Daedalion. Do not edit — rebuild with `daedalion build`.
## Project Domains
- lending-underwriting — use #lending-underwriting skill or ...
## Working with Specs
Specifications are in openspec/. Use .github/skills/ for implementation guidance...
```

### Root cause

`src/generators/instructions.ts` only extracts sections under a `## Conventions`
heading from `project.md`. The heading in this project's `project.md` is
`## Project Conventions` — one word difference, zero match.

```typescript
// instructions.ts — only this exact heading is extracted:
if (/^## Conventions\s*$/.test(line)) {
  inConventions = true;
```

### Impact

For SDK mode this is actually fine: the SDK agent doesn't read
`copilot-instructions.md` at all — context comes from skill files. For IDE
Copilot it is a real regression: the project conventions are invisible to the
IDE assistant.

### Recommendation

Two options:

- **Quick fix**: rename `## Project Conventions` → `## Conventions` in
  `openspec/project.md` to match the extractor pattern.
- **Better fix**: make the heading configurable in `daedalion.yaml`, or extract
  the full `project.md` content (not just one section) as the default, with opt-in
  filtering by heading name.

## Finding 3 — `agents.target: sdk` Has No Effect Today

`daedalion.yaml` supports `agents: target: sdk` but none of the generators
branch on it. `skill.ts`, `instructions.ts`, `agent.ts`, and `tools.ts` all
produce identical output regardless. The config key is parsed and validated but
never consulted during generation.

This matters because SDK and IDE have fundamentally different context-delivery
models:

| | IDE (Copilot Chat) | SDK (Copilot SDK) |
|---|---|---|
| `copilot-instructions.md` | Read by IDE on every turn | **Not used** |
| `skills/SKILL.md` | Loaded when `#skill` referenced | Loaded via `skill_directories` |
| `skills/*.md` spoke files | Navigated via links | **All loaded eagerly** |
| `agents/*.agent.md` | Used as agent persona in IDE | **Not used by SDK** |

### Recommendation

Wire `agents.target` into the generators so that building for SDK produces
output optimised for that consumption model.

## Finding 4 — Tool Stubs Generator Is Unused

`src/generators/tools.ts` can generate Python (or JS) function stubs from the
`tools:` frontmatter in a spec. This was not invoked for this project because
the tools already existed before daedalion was added.

The generated stubs would live at `.github/tools/<tool_name>.py` — not in the
project's actual `tools/` directory. There is currently no way to configure the
output path.

### Recommendation

This feature is most useful at project **bootstrap** (spec exists, code does
not). Add a `tools.output_dir` config key, or document clearly that tool stubs
are starter templates to be moved and edited, not kept in `.github/`.

## Overall Value Assessment

| Area | Value in IDE projects | Value in SDK projects |
|---|---|---|
| OpenSpec spec management (`openspec validate`, change proposals) | ✅ High | ✅ High |
| `SKILL.md` as single-file context | ✅ High | ✅ High (hand-tuned) |
| Per-requirement spoke files | ✅ Good for IDE browsing | ❌ Token duplication |
| `agents/*.agent.md` | ✅ Domain persona for IDE | ❌ Not used by SDK |
| `copilot-instructions.md` | ✅ Project conventions for IDE | ❌ Not used by SDK |
| Tool stub generation | ✅ Bootstrap scaffold | ✅ Bootstrap scaffold |
| `daedalion validate` / drift detection | ✅ High | ✅ High |

**Summary**: Daedalion adds real value as a spec-to-artifact compiler for IDE
projects. For SDK agent projects it is useful primarily for the **openspec
workflow** (spec management, drift detection, change proposals) and for the
initial `SKILL.md` scaffold — but the generated output layout is not yet
optimised for SDK consumption and should not be committed as-is.

## What We Did

1. Ran `daedalion build` — produced 11 files.
2. Reviewed the diff — found hub/spoke duplication (+~2,860 tokens/call) and
   missing project conventions in `copilot-instructions.md`.
3. Reverted `.github/` to the hand-tuned `SKILL.md` (option C).
4. Removed two dead constants (`AGENT_MD_PATH`, `SKILL_PATH`) from `agent.py`
   that referenced generated files the SDK was already loading implicitly.
5. Filed this document as feedback for the daedalion maintainer.

agents:
  target: sdk       # ide | sdk — currently parsed but not acted on

  conventions_heading: "Project Conventions"  # NEW: match any heading name

  output_dir: ./tools  # NEW: emit stubs into project tools dir, not .github/tools
